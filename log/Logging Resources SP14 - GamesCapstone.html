Files and Setup[edit]

There are three files that you will need to integrate the CGS Logging API into your game. All three files are provided in the project directory for your game, which you can find at /projects/instr/14sp/cse481d/<your_game_name> on any CSE machine. Each file is described below:

  • CGSCommon.swc - This swc contains the Logging API. To use this swc in your project in FlashDevelop, create a libs folder outside of your src folder and place the swc in that folder. Right-click on the swc file and select "Add to Library". If this throws a warning message just click "Continue". Then, right-click on the swc file again and select "Options". Make sure that the "included library (include completely)" option is selected. This will add the swc to your project so you can reference all of the classes and methods in the logging API.
  • Logger.as - This wrapper class contains all of the functionality that you will need to log data from your game. Initialize the logger class using your game-specific information (see next bullet point).
  • Logging.txt - This file contains the game-specific information that you should use to initialize the Logger class, as well as the username and password for your database.
  • LoggingExample.as - This is an example class that uses the Logger.as to save some data.
  • getdata_dev.py - This is the script for dumping data into a JSON object for the development server. More information in the Analysis section below.
  • getdata_prd.py - This is the script for dumping data into a JSON object for the production server. More information in the Analysis section below.


Functionality[edit]

The Logger.as class has the following functionality:

  • The Logger constructor will generate a new unique user ID that it will store and use for logging all events associated with that user. It will retrieve that user ID from the cache if a player returns, which will allow you to measure return rate information.
  • The logLevelStart function should be called each time a level starts, passing in a unique qid (level id) as described in the IDs section below.
  • The logAction function can be called to record any information about what the player does during a level. This could be key presses, mouse clicks, or any other high-level information that you might want to know later. The logLevelStart function must be called before the logAction function, and the stored action data will be associated with the currently active qid (level id).
  • The logLevelEnd function should be called a the end of each level, before calling logLevelStart on a new level. Note that if a player quits (closes the browser window) in the middle of a level, the level start and any completed actions will still be logged, even though logLevelEnd is never called.

All three of the logging functions allow you to pass in an arbitrary data object that will be saved along with the start, action, or end information. An example class LoggingExample.as that uses the Logger.as file to store data is included in your project directories as well.


IDs[edit]

The Logging API requires a bunch of different IDs. Here's an overview of what each one means:

  • GID: The id of your game. You can find it in logging.txt in your project space (see Files and Setup section).
  • UID: The user id of the person playing. Automatically generated when the logging library initializes, and fetched from the cache if possible so returning users will have the same ID.
  • CID: The category id of the current version. This is meant to segregate data into groups - Tuesday's playtest is cid 4, but Thursday's playtest is cid 5, making it easy to differentiate the data between both playtests. This should be used like a version id. You should add your cids to the database table 'categories' in your main game database. It's also a good idea to use separate category ids for your debugging version so that your test data doesn't get mixed up with your real data.
  • VID: Just to make it confusing, there's also a version id. Just stick with 1 for all of your prototypes and use the cid to differentiate between versions. We use the vid for vastly different versions of our games (not just tweaks) which is why this exists.
  • QID: The level id. It actually stands for "quest id." No, I don't know why it's "quest" but that's the way the database is already set up, so we're stuck with it. Each level in your game should have unique qid, so you can tell which traces came from which levels.
  • DQID: A trace id. Actually stands for "dynamic quest id." Bascially a unique identifier for a particular instance of a user playing a level. This is automatically generated by the logging library.
  • AID: the action id of any particular action. Players may do things like kill enemies, pick up loot, move to a new location, etc.; each type of action should have its own id. You will have to generate these yourself. Typically you will also want to store detailed data along with the id. It's a good idea to keep a list of your AIDs somewhere so you know which numbers you've used in the past.


Debugging[edit]

For debugging purposes, you can access your data directly from the database.

DO NOT query the database directly in your flash games. This will be absolutely terrible for performance for everyone if you have any bugs. For example, a group in the past had a problem where every instance of the game accidentally queried the logging data for every user frequently. This brought the server to a crawl, and if your game has a bug like this and gets pirated, then you won't be able to fix it.

Use the database to debug your logging and make sure that all of the information you want to record is actually being stored in the database. Each game has three databases each with a ton of tables, many of which you will never need to look at (they are used in some of our other research games). The only ones you need to care about are:

  • In the database cgs_gc_<game_name>:
    • categories: the table is where you should save your category ids (or if you don't want to do this, make sure to keep a secure list of them somewhere).
  • In the database cgs_gc_<game_name>_log:
    • player_pageload_log: Row for each time the game is loaded.
    • player_quests_log: Row for each time you call logLevelStart/End.
    • player_actions_log Row for each time you call logAction.

The columns should be mainly self explanatory if you know what the ids mean. anything with "ts" is a Unix timestamp. Anything with "detail" is your custom data. Anything else can probably be ignored.


To log into the database via the command line, follow these steps:

  • ssh into any CSE machine
  • type "mysql -h dev.db.centerforgamescience.com -u <your_user_name> -p" and then type your password. You can find your username and password in the Logging.txt file in your project directory at /projects/instr/14sp/cse481d/<your_game_name>. This will open the command-line mysql prompt.
  • type "show databases"; to see all the databases
  • type "use database <name>;" to select a database (you'll mostly want to look at cgs_gc_<game_name>_log)
  • type "show tables;" to see all the tables in the databases
  • type queries directly into the command prompt, for example "select * from player_actions_log order by log_ts desc limit 10;" to see the entries in the actions log ordered from most recent to least recent.


Analysis[edit]

For any purpose other than debugging, it's best to access your data as a JSON object and analyze it with python scripts. There are two files in your project directory at /projects/instr/14sp/cse481d/<your_game_name> that you can use to dump your data into a JSON object:

  • getdata_dev.py is for dumping data from the development server. All of your games are currently logging to the development server, so you should use this script while implementing and debugging your game.
  • getdata_prd.py is for dumping data from the production server. There will be more information about setting up your game to log to the production server in the instructions for your first release. We will use the production server for all of your real releases because it is more reliable and robust. Please debug your logging on the development server first to make sure that it's working.

Both of these scripts ask you to enter your game id and category id, for example "python getdata_dev.py <game id> <category id>". The script will dump all of the players, traces, and actions for that game id and category id into a JSON file. This script will only dump the data from that category id, which makes it very easy to separate data from different releases. For example, you could use one category id to debug your game, another for your initial release to your friends, another for your first online release, and another for your Kongregate release.

Unfortunately, CGS has not documented the format of the JSON object anywhere formal, but here's a high level idea. Your JSON object is a collection of players. Each player has a number of properties like pageloads (the number of times the player loaded the game) and levels (level traces with unique dqids). Each level has a set of actions, and each action has properties like aid (the action id for that action), a_detail (any extra information you stored along with the action), and ts (the timestamp for that action). Feel free to explore the data structure yourselves, for example you can type "print player" in Python to see what the JSON data looks like for a whole player (it will look messy :) If you have any specific questions about the JSON object let me know!

Parsing the Dump Script in Python[edit]

Even though you ran some Python to generate the JSON dump (to fetch it from the CGS server), feeding that JSON back into python seems like a good way to explore your data. It turns into a nice python struct easily. 

import json, sys
from datetime import datetime

if len(sys.argv) < 2:
    print "enter json file"
    sys.exit(0)
    
f = open(sys.argv[1]);
players = json.loads(f.read())

print "Number of players: %d" % len(players) 

for player in players: 
    print ""
    print "Player: %s" % player['uid']
    print "\tPage loads: %d" % len(player['pageloads'])
    print "\tLevels logged: %d" % len(player['levels'])
    
    levels = player['levels']
    for level in levels:
        print "\t\tLevel/'Quest' id: %s (%s)" % (level['qid'], datetime.fromtimestamp(level['log_q_ts']))
       
        actions = level['actions']
        for action in actions:
            print "\t\t\tAID: %s, Timestamp: %s, Action details: %s" % (action['aid'], action['ts'], action['a_detail'])
